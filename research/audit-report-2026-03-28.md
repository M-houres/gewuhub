# Gewu 项目客观审计报告（2026-03-28 ~ 2026-03-29）

## 1. 审计范围与方法
本报告基于当前代码仓库 `D:\gewu` 的实际实现进行客观审计，覆盖账号体系、积分计费、任务处理链路（降重/降AIGC/AIGC检测）、文件处理、后台管理、安全控制、可部署性与测试可验证性。  
审计依据不是主观体验，而是构建与冒烟测试结果（`npm run build`、`npm run smoke:critical`）以及核心代码路径核查（`apps/api`、`apps/admin`、`apps/web`、`apps/worker`）。+
## 2. 当前已完成能力（可验证）

### 2.1 账号与权限
已实现注册、登录、登出、邮箱验证、忘记密码、重置密码完整链路。  
未验证邮箱用户会被拒绝登录（403 + `EMAIL_NOT_VERIFIED`），登录态失效后可回退到登录流程，不是白屏。  
用户接口与管理接口已隔离，管理端采用独立管理员鉴权。+
### 2.2 积分与计费（服务端主导）
积分扣减发生在服务端，任务创建前先做余额校验。  
任务失败或取消触发退款，且有一次性防重保护。  
并发扣费场景通过用户级锁控制，已覆盖并发测试，避免“多扣/少扣”。  
积分变动、管理员手工加减积分、关键后台操作均有日志记录，可审计。+
### 2.3 核心任务能力
`reduce-repeat`、`reduce-ai`、`detect` 三条主链路可跑通。  
支持 5 平台维度：`cnki`、`weipu`、`paperpass`、`wanfang`、`daya`。  
检测任务支持 PDF 报告生成与签名下载链路。  
降重与降AIGC已统一使用共享改写主流程（你要求的“同一套规则基础”已落地）。+
### 2.4 15 组规则配置 + 三模式执行
后台可配置 15 个槽位（3任务类型 × 5平台）的规则参数。  
每个槽位可独立配置执行模式：`rules_only`、`hybrid`、`llm_only`。  
当模型不可用时会安全回退 `rules_only`，避免业务中断。  
当模型在后台配置了 API Key 时，模式可保持在 `hybrid/llm_only`（不再强制回退）。+
### 2.5 文件处理与下载
上传有类型白名单与大小限制。  
下载链路已支持签名校验与一次性票据。  
DOCX 任务有本地兜底链路、Worker 回调链路、进度轮询链路；回调完成后保留实际输出地址，不再被错误覆盖。  
文档类任务存在重计费逻辑：按提取后的真实内容重算，支持补扣/退款并留痕。+
### 2.6 后台管理能力
已具备用户、任务、订单、套餐、模型、积分、内容、系统设置等核心管理接口。  
工作台左侧导航开关可在后台控制显示/隐藏。  
模型 API Key 可后台配置与清除，`hasApiKey` 状态同步可见。+
### 2.7 自动化验证状态
`apps/api`、`apps/admin` 构建通过。  
整套 `smoke:critical` 已通过（包含认证、权限、并发扣费、任务安全、PDF 报告、DOCX 回调、支付退款模拟、后台设置、执行模式路由等 20+ 脚本）。+
## 3. 仍未完成或部分完成（真实差距）

### 3.1 真实大模型适配层（关键未闭环）
当前 `ModelRouter` 仍是占位路由，尚未接入真实 provider SDK/HTTP 调用。  
因此 `hybrid/llm_only` 在“有 Key”时可保持模式，但推理内容仍主要由本地规则/占位逻辑生成。  
结论：执行策略框架已完成，真实推理能力尚未完成。+
### 3.2 真实支付网关
已实现订单、回调、退款、幂等处理的技术骨架与测试，但当前仍是 mock/模拟链路。  
未接入真实 Stripe/支付宝/微信支付商户配置与正式验签密钥。  
结论：支付业务流程框架可用，生产收款能力未正式上线。+
### 3.3 平台算法深度
当前可做到“平台可配置 + 可执行 + 可升级”，但 15 组规则在“深度还原各平台官方检测器行为”上仍有差距。  
其中 CNKI/PaperPass 规则覆盖相对更完整，其余平台仍需持续补充规则包与样本校准。+
### 3.4 前端体验收口
核心流程可跑，但你此前点名的一些产品化细节（批量处理体验、历史记录呈现一致性、中英文统一等）仍需继续收口。+
## 4. 代码与业务风险评估

### 4.1 代码风险（中）
仓库处于多轮迭代后的“脏工作区”，存在并行开发痕迹。  
建议上线前做一次“最小变更集整理 + 编码巡检 + 无效日志清理”，避免脏改动进入生产。+
### 4.2 业务风险（中高）
若在“真实模型 + 真实支付”未接入前直接商业化上线，会出现“流程可用但核心商业能力不完整”的风险。  
建议先灰度上线规则版，明确标注能力边界，并在模型/支付接入后切到正式模式。+
### 4.3 运维风险（中）
需确认生产环境的 SMTP、OSS 生命周期策略、域名证书与反向代理配置。  
建议同时接入错误监控告警（Sentry 已有依赖基础，可补环境配置）。+
## 5. 当前结论
项目已经达到“核心链路可运行、可测试、可继续压实上线”的阶段，不是空壳。  
但若以“完整商业化版本”定义，仍缺两块关键闭环：  
1) 真实大模型适配；2) 真实支付网关接入。  
其余功能（账号、积分、任务、后台、下载、安全校验、并发控制）整体已经具备上线前收口基础。+
