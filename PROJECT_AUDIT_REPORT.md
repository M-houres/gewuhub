## 一、项目概览

### 1.1 项目架构
这是一个基于微服务架构的学术文本处理平台，主要功能包括：
- 学术论文查重检测（CNKI V3算法）
- AI文本改写降重
- 文档处理（DOCX/PDF/TXT）
- 用户积分系统
- 支付订单系统
- 多AI模型路由

### 1.2 技术栈
**前端**:
- Web端: Next.js 14.2.5 + React 18 + TailwindCSS
- Admin端: Vite + React 18 + Ant Design 5.29.3

**后端**:
- API: Fastify 5.8.4 + TypeScript
- Worker: BullMQ 5.71.1 后台任务队列
- 数据库: PostgreSQL 16 + Prisma ORM
- 缓存/队列: Redis 7

**部署**:
- Docker Compose
- Nginx反向代理
- 健康检查机制

### 1.3 代码规模
- API核心代码: ~10,443行TypeScript
- 4个独立服务容器
- 完整的CI/CD部署脚本

## 二、关键问题与风险

### 🔴 严重问题

#### 2.1 安全漏洞

**问题1: 默认密钥未更改**
- `.env.example`中包含弱密钥: `change-me-admin-token`, `change-me-admin-password`
- 生产环境必须替换所有`change-me-*`占位符
- 风险: 未授权访问管理后台

**问题2: 密码存储**
- 需确认密码是否使用bcrypt/argon2等安全哈希
- 检查`apps/api/src/auth.ts`的密码验证逻辑

**问题3: SQL注入风险**
- 使用Prisma ORM，基本安全
- 但需检查动态查询构建部分

**问题4: 文件上传安全**
- 限制文件类型: `.docx`, `.pdf`, `.txt`
- 最大10MB限制
- ⚠️ 需要添加文件内容验证，防止恶意文件

#### 2.2 性能瓶颈

**问题1: 单体server.ts文件过大**
- `server.ts`: 143,163行代码
- 所有路由都在一个文件中
- 建议: 拆分为模块化路由

**问题2: 缺少数据库索引优化**
- Prisma schema中只有基本索引
- 高频查询字段需要复合索引
- 建议添加: `User(email)`, `Task(userId, status)`, `Order(userId, createdAt)`

**问题3: Redis缓存策略不明确**
- Redis用于队列和持久化
- 未见明确的缓存层设计
- 建议: 添加热点数据缓存（用户信息、系统配置）

#### 2.3 代码质量问题

**问题1: 错误处理不完善**
- 缺少全局错误处理中间件
- API错误响应格式不统一
- 建议: 统一错误码和错误消息格式

**问题2: 日志系统不完整**
- 使用Fastify内置logger
- 缺少结构化日志
- 建议: 集成Winston或Pino，添加请求追踪ID

**问题3: 测试覆盖率为0**
- 项目中没有任何测试文件
- 无单元测试、集成测试
- 风险: 重构和上线风险极高

## 三、算法模块分析

### 3.1 CNKI查重算法 (cnki-detect-v3.ts)

**优点**:
- 实现了多层次检测: 词汇、短语、结构
- 风险等级分级: none/low/medium/high
- 考虑了上下文密度加权

**问题**:
- 算法硬编码在代码中，难以调整
- 缺少算法版本管理
- 建议: 将规则配置化，支持动态更新

### 3.2 改写引擎 (rewrite-engine.ts)

**优点**:
- 支持多平台适配（PaperPass、维普等）
- 词汇替换策略完善
- 保护关键术语

**问题**:
- 改写规则过于简单，容易被识别
- 缺少语义理解
- 建议: 集成真实AI模型API进行深度改写

## 四、数据库设计审查

### 4.1 Schema分析

**表结构**:
- User: 用户表（支持邮箱/手机登录）
- Task: 任务表（查重/改写任务）
- ModelConfig: AI模型配置
- Plan: 套餐计划
- Order: 订单表
- PointRecord: 积分记录

**优点**:
- 使用CUID作为主键，安全性好
- 时间戳字段完整（createdAt/updatedAt）
- 关系定义清晰

**问题**:
- 缺少软删除字段（deletedAt）
- 缺少数据审计字段（操作人、IP等）
- Task表的payload和result使用Json类型，查询性能差
- 建议: 添加索引、分表策略（Task表会快速增长）

### 4.2 数据迁移

**问题**:
- 使用`prisma db push`而非`prisma migrate`
- 缺少迁移历史管理
- 建议: 切换到正式的migration工作流

- 安全头: X-Frame-Options, X-Content-Type-Options, CSP
- 静态资源缓存

**任务3: 环境变量安全**

在`schema.prisma`中添加：
```prisma

  @@index([email])
  @@index([phone])
}

model Task {
  // ... 现有字段
  @@index([userId, status])
  @@index([createdAt])
}

model Order {
  // ... 现有字段
  @@index([userId, createdAt])

```

**任务5: API响应缓存**
